## 项目信息

这是一个名为 **死神vs火影** 的 2D 格斗游戏。使用 ActionScript 3 语言编写，Adobe AIR 作为运行时。

项目整体采用 **代码 + 美术** 分离开发：
代码技术栈主要使用 Intellij IDEA + Flash/Flex 插件进行开发，同时也兼容 VSCode（或基于其的其他智能编辑器） + ActionScript & MXML 插件的形式开发。
美术技术栈使用 Adobe Animate 进行导出 `BleachVsNaruto_FlashSrc` 目录下的 Flash 源文件。

## 项目核心架构及简要说明

- `BleachVsNaruto_FlashSrc` 游戏所有美术资源存放目录，通常情况下，这里仅涉及美术开发，除非特殊要求，在进行代码开发的时候，请尽量避免在此目录下进行长时间任务。
- `CORE_Components` 游戏的可交互组件，其作用是作为 `BleachVsNaruto_FlashSrc/SRC_XFL/#swc/component/component.xfl` 文件的代码类资源使用，它会编译成可供 Adobe Animate 中可使用的 **组件库**。它的使用频率较低，基本是面向使用 Adobe Animate 进行人物或 UI 编辑时才会使用。目前开发重心并不在此，常规游戏性内容开发也不涉及此处。
- `CORE_KernelLogic` 游戏的核心逻辑，此模块是项目的重点，存放了所有关于游戏核心玩法（例如角色战斗，游戏玩法）的逻辑，建议详细阅读此模块，充分理解该模块的所有内容，生成足够使用的上下文，游戏开发 90% 都会聚焦在此模块。
- `CORE_Shared` 游戏的共享组件，对外接口，存放游戏需要对外暴露的部分常量和共用方法，同时也存放游戏的版本记录，不同版本的 `CORE_Shared` 组件可以通过检查版本来确保版本一致。该模块同时也是 `BleachVsNaruto_FlashSrc/SRC_XFL/#swc/shared/shared.xfl` 文件的代码资源类，在编译 BleachVsNaruto_FlashSrc 的资源时会自动导出到`shared/lib/swc/ide/shared.swc` 文件。
- `CORE_Utils` 游戏的部分公用工具，将部分常用方法提取，供项目的其他模块调用。

## 代码风格

请确保生成的代码与项目整体的代码风格保持一致。以下是从 `.idea/codeStyles/Project.xml` 解析出的详细代码风格规范：

### 引号与分号

- **单引号优先**：字符串统一使用单引号 `'`，不使用双引号 `"`
- **强制分号**：所有语句必须以分号 `;` 结尾

### 大括号与换行

- **强制大括号**：所有控制流语句（if/while/for/do-while）必须使用大括号，即使只有一行语句
- **控制关键字换行**：else、catch、finally 关键字单独起一行
- **括号换行**：方法调用、方法定义、数组初始化等的左右括号各占一行

### 对齐与缩进

- **对齐对象属性**：对象字面量属性对齐
- **对齐变量声明**：变量声明对齐
- **对齐多行链式调用**：链式方法调用对齐
- **对齐多行参数**：方法调用和定义的多行参数对齐
- **对齐多行操作符**：二进制、三元操作符换行后对齐
- **case 不缩进**：switch 语句中的 case 不额外缩进

### 换行与折行

- **对象字面量换行**：对象字面量换行时左右大括号各占一行
- **赋值换行**：赋值语句较长时等号后换行
- **注释换行**：长注释自动换行
- **数组初始化换行**：数组初始化换行时左右大括号各占一行

### 其他规范

- **C风格注释格式化**：自动格式化块注释
- **行分隔符**：使用 CRLF（Windows 风格）作为行分隔符

通常项目负责人会使用 IntelliJ IDEA 对代码进行格式化，请遵循以上规范。

### 注释规范

#### 1. ASDoc 文档注释

**基本要求**：

- 所有 public/protected 方法、类、属性必须添加完整的 ASDoc 注释
- **优先遵循 ASDoc 规范**；ActionScript 无对应能力时再参考 JavaDoc
- 包含 `@param`、`@return`、`@throws` 等标签（如适用）
- 注释正文使用**中文**；保留文件中已有的 `@author`、`@version` 等元数据
- **仅补文档时不改业务逻辑**（不删改实现、不清理 dead code，除非用户另行要求）

**详细注释标准**（参考 `CheatCodeManager.as`）：

**类注释**：

- **功能概述**：简要说明类的用途和职责
- **核心设计理念**：列出 3-4 个关键设计原则（如单一职责、全局共享、低耦合、高性能）
- **设计特点**：列出 4-5 个关键特性（如共享监听器、双重触发模式、生命周期管理、高效存储、数据封装）
- **工作原理**：分阶段说明工作流程（5-6 个阶段，如注册阶段 → 监听阶段 → 匹配阶段 → 触发阶段 → 清理阶段）
- **使用示例**：提供 2-3 个实际使用场景的代码示例（类级优先使用 `@example` + `<pre>`）
- **性能考量**：列出 3-4 个性能优化点
- **元数据**：包含 `@see`、`@author`、`@version` 标签

**方法注释**：

- **功能概述**：简要说明方法的功能和作用
- **工作流程**：对于复杂方法，分步骤说明执行流程（如 5-7 个步骤）
- **格式说明**：对于涉及格式的方法，详细说明格式规则（如分隔符、按键名称、大小写等）
- **模式说明**：对于支持多种模式的方法，说明每种模式的特点（如重复触发、单次触发）
- **使用示例**：提供 1-2 个实际使用场景的代码示例
- **注意事项**：列出 2-3 个使用时需要注意的事项
- **参数说明**：使用 `@param` 标签详细说明每个参数的用途、类型、默认值
- **返回值说明**：使用 `@return` 标签说明返回值的类型、含义、可能的异常情况
- **相关引用**：使用 `@see` 标签引用相关的方法、类、属性

**属性注释**：

- **功能说明**：简要说明属性的用途
- **约束条件**：说明属性的取值范围、默认值、约束条件
- **使用场景**：说明属性的使用场景和注意事项

**辅助类注释**（内部类、`private` 嵌套类等）：

- **设计目的**：列出 3-4 个设计目的（如数据封装、状态追踪、性能优化、类型安全）
- **状态流转**：分阶段说明状态变化（如初始状态 → 匹配中 → 触发成功 → 重置状态）
- **字段说明**：每个字段都包含详细的用途说明和约束条件
- **构造函数说明**：说明构造函数的参数要求和初始化逻辑
- 类定义处添加 `@private`，避免出现在公开 API 文档中

**接口与事件类注释**：

- **接口**：说明契约职责，使用 `@see` 指向典型实现类
- **独立事件类**：为事件类型常量补充 `@eventType` 与分派时机说明

**ASDoc 专有标签与惯例**：

| 标签 / 机制 | 用途 |
|------------|------|
| `[Event]` 元数据 | 在类定义前声明可分派事件（Flex/AIR 组件文档惯例） |
| `@eventType` | 标注事件类型；`[Event]` 注释块指向常量（如 `ClassName.EVENT_XXX`），常量定义处指向事件名字符串（如 `load_complete`） |
| `@example` | 代码示例，常与 `<pre>` 组合；类级示例优先于仅使用 `<p><b>使用示例</b></p>` |
| `@default` | 标注 public 属性的默认值 |
| `@private` | 标注私有字段、私有方法、内部类，以及仅需 `@private` 的 setter |

**事件常量描述**（Adobe 标准句式）：

```actionscript
/**
 * EVENT_LOAD_COMPLETE 常量定义了
 * <code>load_complete</code> 事件对象的 {@link flash.events.Event#type} 属性值。
 *
 * @eventType load_complete
 */
public static const EVENT_LOAD_COMPLETE:String = 'load_complete';
```

**事件类 `[Event]` 声明示例**：

```actionscript
/**
 * 内容加载完成时分派的事件。
 *
 * @eventType SuperPlayer.EVENT_LOAD_COMPLETE
 */
[Event(name='load_complete', type='flash.events.Event')]
public class SuperPlayer extends Sprite { ... }
```

**getter / setter 对**：

- **getter**：完整 ASDoc（功能、约束、`@return`、`@default` 等）
- **setter**：仅 `/** @private */`，不重复 getter 说明

**链接与引用规范**：

- 公开文档中**不要**对私有成员使用 `{@link}`（如 `_url`、`playVideo`）
- 类级文档中**不要**直接引用私有字段名
- 跨类引用内部类时避免 `{@link InsVideo}`，改用文字说明或 Flash API 全限定名
- 方法引用使用 `@see #methodName()`（带括号）

**注释详略分级**：

- **复杂管理器 / 组件**（如 `CheatCodeManager`、`SuperPlayer`）：类级含设计理念、设计特点、工作原理、示例、性能考量
- **算法 / 编码类**（如 `JPGEncoder`）：重点文档化 public API；私有实现方法仅标 `@private`
- **简单工具类**：可省略与职责无关的章节，避免重复堆砌

**非 ActionScript 文件**：

- JavaScript 桥接脚本（如 `AS.js`）使用 **JSDoc**，不适用 ASDoc

**批量补全 ASDoc 任务**：

- 按目录或文件依次处理 public/protected 及必要的 internal 成员
- 完成后**简短报告**即可（如「`XXX` 的 ASDoc 注释已补全」），不必展开逐条列举

#### 2. 逻辑注释

**基本要求**：

- 复杂逻辑必须添加行内注释，说明代码意图
- 关键业务逻辑必须添加注释
- 非常规的算法或实现方式必须添加注释说明

**详细注释标准**：

**行内注释**：

- 在关键代码行上方添加注释，说明代码意图
- 注释应该简洁明了，直接说明"做什么"而非"怎么做"
- 对于复杂的条件判断，说明判断的目的

**代码块注释**：

- 对于代码块（如 if、for、while），在块开始处添加注释说明块的用途
- 对于多行逻辑，在块开始处添加注释说明整体逻辑

**注释格式**：

- 使用 `//` 注释单行与块逻辑，但不要注释局部变量或类属性
- 使用 `/** */` 注释多行文档，使用此注释局部变量或类属性
- 注释应该与代码对齐，保持代码整洁

**注释示例**：

```actionscript
/** 名称 */
var name:String = 'name';

// 检查条件
if (name == 'name') {
    // 具体执行块逻辑
    name = '123';
}

// 输出
trace(name);
```

- 段落之间空一行
- 行首不缩进
- 长段落适当分行，但保持语义连贯

**列表格式**：

- 无序列表使用 `-` 作为标记
- 有序列表使用 `1. 2. 3.` 作为标记
- 列表项之间空一行（简短项可紧接）
- 嵌套列表缩进 4 个空格

**列表前空行**：粗体标记的小标题（如"**列表格式**："）与下方列表之间需要空一行。

**CHANGELOG 分类**：

```markdown
### 新增
- 新增内容1
- 新增内容2

### 更改
- 更改内容1

### 修复
- 修复内容1

### 移除
- 移除内容1
```

### 3. 链接与引用

**链接格式**：

- 内联链接：`[显示文本](文件路径或URL)`
- 引用链接：在文件末尾定义
- 引用链接格式：`[链接名称]: URL`

**链接示例**：

```markdown
详细信息请参阅 [CHANGELOG.md](CHANGELOG.md)。

下载 SDK：[flex4.16.1-air51.0.1.1](https://github.com/...)

[flex4.16.1-air51.0.1.1]: https://github.com/...
[Intellij IDEA 2022.1.4]: https://download.jetbrains.com/idea/...
```

**相对路径**：链接到项目内文件使用相对路径，避免使用绝对路径。

### 4. 代码与强调

**代码格式**：

- 行内代码使用反引号：`` `代码` ``
- 代码块使用三个反引号，指定语言：

  ```markdown
  ```actionscript
  // 代码内容
  ```
  ```

**强调格式**：

- 粗体：`**文本**` 或 `***文本***`
- 斜体：`*文本*`
- 高亮关键术语：如 **必须**、**推荐**、**可选**

### 7. CHANGELOG 格式

**版本格式**：`## [版本号] - 日期`

- 日期格式：YYYY-MM-DD
- 版本号：遵循项目已有格式

### 8. 行分隔符

**使用 CRLF**：所有 Markdown 文件使用 Windows 风格的 CRLF 行分隔符。

### 9. 开头说明

**简短开头**：每个文档开头应有 1-2 句话简要说明文档用途。

```markdown
# 更新日志

本文件的所有重要变更都将会记录在此文件中。
```

### 10. 目录生成

**可选目录**：对于较长文档，可添加目录，使用 `<!-- omit in toc -->` 标记不需要列入目录的标题。

```markdown
## 目录 <!-- omit in toc -->

- [工程状态](#工程状态)
- [下载二进制游戏](#下载二进制游戏)
- [更新日志](#更新日志)
```
